# 一页纸需求理解说明

## 一句话任务定义

为某大型环卫集团设计一个“事件调度与管理看板”MVP 交互原型，让高管层快速看清各片区运行状态，让调度/片区主管持续追踪事件流转，并通过 AI 辅助判断下一步调度动作。

## 对环卫调度业务痛点的理解

环卫调度的核心难点不在于单个事件是否能被记录，而在于多片区、多事件、多角色同时运转时，管理者很难快速形成统一判断。高管层需要知道当前哪些片区异常、异常程度如何、是否影响整体运行；调度/片区主管则需要知道具体事件卡在哪个节点、谁正在负责、是否超时、下一步应该派人、派车、复核还是升级。

因此，这个原型需要同时解决“宏观态势不清”和“微观流转不明”两个问题：前者通过片区总览建立整体判断，后者通过事件详情和流转节点明确执行责任。

## 原型设计理念

1. 突出总览：首页优先呈现片区状态、异常数量、待处理事件和关键提醒，让高管层能在第一屏完成态势判断。
2. 强调流程：从片区进入事件列表，再进入事件详情，围绕“在哪、谁负责、下一步做什么”组织信息。
3. 引入 AI：AI 不替代人工决策，而是在事件详情中给出风险判断、推荐动作、推荐资源和原因解释，帮助调度主管更快形成处理方案。

## 原型如何解决痛点

- 针对片区状态分散：通过首页总览把多个片区的运行状态集中展示，并用颜色、标签和指标区分正常、预警、异常、处理中等状态。
- 针对事件优先级不清：在事件列表中展示事件类型、位置、状态、优先级、负责人和截止时间，帮助主管先处理更紧急的事件。
- 针对责任和动作不明确：在事件详情中固定展示负责人、当前节点、下一步动作和处理时限，减少沟通负担。
- 针对调度判断依赖经验：用 AI 建议区补充判断依据和推荐动作，让调度过程更可解释、更一致。

## 核心交互闭环

```text
首页片区总览
-> 发现异常片区
-> 查看片区事件列表
-> 打开事件详情
-> 查看 AI 建议
-> 确认或调整下一步操作
-> 回到总览观察状态变化
```

这个闭环对应两类角色的连续工作方式：高管层在首页完成全局判断，调度/片区主管在下钻页面完成事件推进，AI 建议作为辅助信息嵌入具体事件处理节点。

## 设计边界

- 本次交付聚焦 MVP 原型，不追求完整业务系统。
- 使用模拟数据表达片区、事件、人员、车辆和 AI 建议，不接入真实生产接口。
- AI 只展示辅助判断方式，包括风险等级、判断原因、推荐动作和推荐资源，不设计真实模型训练或算法实现。
- 原型重点覆盖“总览 -> 下钻 -> 决策 -> 反馈”的主路径，不展开复杂权限、排班、统计报表和历史追溯等后台能力。
- 所有页面和说明围绕 Figma / 墨刀等设计原型交付，确保评审者可以直接查看和演示。
