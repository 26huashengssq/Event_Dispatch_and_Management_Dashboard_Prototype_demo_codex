# 一页纸需求理解说明 v1.2.0

## 一句话任务定义

为大型环卫集团设计一个"事件调度与管理看板"MVP 交互原型，让高管层快速看清各片区运行状态，让调度/片区主管持续追踪事件流转，并通过 AI 辅助判断下一步调度动作。

## 业务痛点理解

环卫调度的核心难点不在于单个事件能否被记录，而在于**多片区、多事件、多角色同时运转**时，管理者很难快速形成统一判断：

| 痛点 | 层面 | 表现 |
|---|---|---|
| 宏观态势不清 | 高管层 | 不知道当前哪些片区异常、异常程度如何、是否影响整体运行 |
| 微观流转不明 | 调度/片区主管 | 不知道具体事件卡在哪个节点、谁负责、是否超时、下一步该派谁 |

因此，本原型需要同时解决"看不清全局"和"追不紧细节"两个问题。

## 三大功能模块如何解决痛点

### 📊 片区总览 → 解决"宏观态势不清"

首屏展示 **KPI 摘要栏**（事件总数 / 待处理 / P0 紧急 / 超时 / 异常片区数）和 **4 个可交互片区卡片**，用颜色标签区分正常、预警、异常、处理中四种状态。高管层在首页即可完成全局判断：哪片出问题、风险多高、资源是否紧张。点击片区卡片可筛选该片区的事件列表，实现"总览 → 下钻"。

### 🔧 事件流转 → 解决"微观流转不明"

以 **5 阶段流转节点**（待派单 → 协调中 → 处理中 → 待复核 → 已关闭）可视化所有事件的分布。点击任一节点筛选该阶段事件，点击具体事件打开详情面板。详情面板固定展示：所属片区、位置、负责人、当前状态、下一个动作、处理时限。底部提供**操作面板**（确认推进 / 改派他人 / 升级处理）供调度主管推进事件。面板内嵌"迷你流转条"直观标注当前所处阶段。详情面板还提供**跳转 AI 辅助**入口，一键定位到该事件对应的调度建议。

### 🤖 AI 辅助 → 解决"调度判断依赖经验"

AI 模块提供两类能力：

1. **风险态势总览**：四象限风险卡片（高/中/低风险未处理 + 已采纳），一眼看清全局 AI 判断分布
2. **逐事件调度建议**：每条建议含风险等级、置信度进度条、判断原因、推荐动作、推荐资源，并提供采纳/调整/暂存操作

**v1.2.0 核心升级**：新增 `POST /api/ai/generate` 后端代理接口，调用外部 OpenAI 兼容 API 实时生成建议。提示词模板自动注入片区状态、事件详情、可用资源等上下文。未配置 API Key 时自动降级为本地静态建议（`api/aiSuggestions.json`），保证原型始终可演示。

## 核心交互闭环

```
首页片区总览（高管全局判断）
  → 发现异常片区 → 点击片区卡片
  → 查看该片区事件列表（按 P0/P1/P2 排列）
  → 点击具体事件 → 查看事件详情（卡在哪/谁负责/下一步）
  → 查看 AI 建议（风险等级 + 推荐动作 + 推荐资源）
  → 确认或调整操作（采纳/调整/暂存）
  → 回到总览观察状态变化
```

实际产品中以 **3 个 Tab 标签页**承载上述闭环：
- Tab 1「📊 片区总览」：KPI + 片区卡片 + 事件列表
- Tab 2「🔄 事件流转」：流转节点 + 事件列表 + 详情面板 + 操作面板
- Tab 3「🤖 AI 辅助」：风险态势 + 建议卡片 + 实时生成 + fallback

跨模块导航通过 `app:navigate` 自定义事件实现（如事件详情 → AI 辅助）。

## v1.2.0 AI 外部 API 接入说明

### 提示词设计思路

后端 `buildPrompt()` 函数根据事件 ID 动态组装提示词，包含三个信息层：
- **片区状态层**：片区名称、当前状态、今日事件数/待处理数/超时数、可用资源
- **事件详情层**：事件 ID、类型、位置、优先级、当前状态、负责人、下一步、处理时限、是否超时、上报人与时间
- **输出约束层**：要求严格返回 JSON，字段包括 riskLevel、confidence、reasons、recommendedAction、recommendedResources

### API 代理架构

```
前端 fetch → POST /api/ai/generate { eventId }
  → server.js handleAiGenerate()
    → loadAiContext(eventId)：读取片区+事件+静态建议
    → buildPrompt(context)：组装提示词
    → fetch(OPENAI_BASE_URL)：调用外部 AI API（Bearer Token 认证）
    → normalizeAiSuggestion()：统一输出格式
  → 前端渲染建议卡片

失败路径：无 Key / 网络异常 / 返回非 JSON
  → buildFallbackSuggestion()：降级为本地静态建议
  → source: "fallback"，附带 fallbackReason
```

### 环境变量

| 变量 | 必填 | 默认值 | 说明 |
|---|---|---|---|
| `OPENAI_API_KEY` | 是 | 无 | 外部 AI API Key，仅后端读取 |
| `AI_MODEL` | 否 | `gpt-4o-mini` | Chat Completions 模型名 |
| `OPENAI_BASE_URL` | 否 | `https://api.openai.com/v1/chat/completions` | 兼容接口地址 |

## 设计边界

- 聚焦 MVP 原型，不追求完整业务系统
- 使用模拟数据（4 片区 / 8 事件 / 5 流转阶段 / 4 AI 建议），数据源为 `data/*.csv`，由脚本生成 `api/*.json`
- AI 仅展示辅助判断方式，不设计真实模型训练；未配置 Key 时自动 fallback，保证演示不中断
- 采纳/调整/暂存操作为前端模拟反馈，不持久化到后端
- 覆盖"总览 → 下钻 → 决策 → 反馈"主路径，不展开权限、排班、历史报表等功能
- Node.js 原生 `http` 模块零依赖，前端 Vanilla JS (ES Module) 无框架
