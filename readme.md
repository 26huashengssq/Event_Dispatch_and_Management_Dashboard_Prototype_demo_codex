# 事件调度与管理看板交互原型需求拆解

## 项目简介

本项目是为某大型环卫集团设计的“事件调度与管理看板”MVP 交互原型，面向高管层和调度/片区主管两个核心角色，帮助他们在同一个看板中完成“看清片区状态、定位异常事件、明确负责人、判断下一步动作”的核心管理闭环。

原型建议使用 Figma / 墨刀完成，重点不是搭建真实系统，而是通过少量高质量页面、清晰交互路径和模拟业务数据，证明该看板具备可理解、可演示、可落地的产品雏形。

## 项目目标

1. 总览片区状态：让高管层快速判断各片区运行是否正常，识别异常区域、风险等级和待处理压力。
2. 追踪事件流转：让调度/片区主管清楚知道事件卡在哪、谁负责、当前进度如何、下一步该做什么。
3. AI 辅助调度：用可解释的 AI 建议辅助判断事件优先级、推荐处理动作和匹配调度资源，但保留人工确认。

## 原型查看 / 运行指南

1. 打开 `prototype/3.交互原型访问链接.txt`，复制其中的 Figma / 墨刀预览链接。
2. 使用浏览器打开链接，确认已开启“可查看”或“可演示”权限。
3. 进入原型后，从“管理看板首页”开始演示，不需要安装本地环境。
4. 建议使用桌面端浏览器查看，推荐视口约为 1440 x 900，以保证看板信息密度和页面层级完整呈现。
5. 如果现场网络不稳定，使用 `recording/4.原型演示录屏.mp4` 作为兜底演示材料。

## 推荐演示路径 / 故事线

1. 背景开场：某大型环卫集团需要同时管理多个片区，管理层需要快速知道哪里异常，调度主管需要知道事件如何推进。
2. 总览发现异常：进入首页，看见 4-6 个片区的运行状态、异常数量、超时事件和风险等级。
3. 锁定重点片区：点击异常片区，查看该片区下的事件卡片列表，识别高优先级事件。
4. 下钻事件详情：打开某个事件，查看事件位置、类型、负责人、当前状态、处理时限和下一步动作。
5. 查看 AI 建议：展示 AI 对风险等级、推荐动作、推荐资源和判断原因的辅助建议。
6. 人工确认调度：调度/片区主管确认或调整 AI 建议，形成下一步处理安排。
7. 回到总览收束：说明该 MVP 已覆盖“高管总览 -> 主管下钻调度 -> AI 辅助判断”的核心闭环。

| 字段 | 示例 | 用途 |

## AI 建议展示方式

AI 建议应作为事件详情中的辅助判断区出现，不单独喧宾夺主。建议采用“结论 + 原因 + 动作 + 人工确认”的展示结构：

- 风险等级：例如“高风险”，用颜色和标签突出。
- 判断原因：列出 2-3 条可解释依据，例如“核心商圈片区”“同类事件近 2 小时重复出现”“当前待处理事件较多”。
- 推荐动作：给出下一步处理建议，例如“优先派东区一组前往处理，并安排车辆支援”。
- 推荐资源：展示建议人员、车辆或班组，不要求真实算法计算。
- 人工确认：保留“采纳建议”“调整方案”“暂不处理”等交互入口，体现 AI 只是辅助决策。

## 1. 一句话任务定义

为某大型环卫集团设计一个“事件调度与管理看板”交互原型，让高管层可以快速看清各片区运行状态，让调度/片区主管可以定位事件、明确负责人和下一步动作，并体现 AI 在第一阶段辅助判断或调度的价值。

## 2. 必做功能清单

> 以下按题目原文顺序拆解。最终交付建议定位为 Figma / 墨刀等设计工具中的可演示交互原型，而不是完整代码系统。

### 2.1 片区运行状态展示屏-让管理层快速看清当前各片区运行状态

- 功能描述：在看板首屏展示各片区的运行状态，例如正常、预警、异常、处理中、已恢复等。
- 输入：片区列表、片区状态、关键指标数据，如事件数量、待处理数量、超时数量、人员/车辆状态。
- 输出：可视化的片区状态总览，支持高管层一眼识别高风险片区和整体运行态势。
- 运行方式：打开 Figma / 墨刀原型首页后直接进入管理看板，通过地图、分区卡片、状态矩阵或列表查看各片区状态。
- 验收标准：无需阅读说明即可判断哪些片区正常、哪些片区异常、异常程度如何；状态颜色、标签和指标含义清晰一致。

### 2.2 事件全流程屏-让管理人员知道事件卡在哪、谁负责、下一步该做什么

- 功能描述：设计事件卡片和事件详情流转，让调度/片区主管能定位事件所属片区、当前节点、负责人、优先级和下一步处理动作。
- 输入：事件列表、事件位置、事件类型、负责人、处理状态、流转节点、建议动作。
- 输出：事件卡片、事件详情页或侧边抽屉，清楚呈现“在哪、谁负责、下一步做什么”。
- 运行方式：在看板中点击片区或事件卡，进入事件详情；可通过筛选、搜索或状态分组定位事件。
- 验收标准：评审者点击任一事件后，可以在 5 秒内找到事件位置、负责人和下一步动作；不同状态的事件有明确视觉区分。

### 2.3 智能辅助调度机制-体现 AI 在第一阶段可以如何辅助判断或调度

- 功能描述：在原型中设计 AI 辅助模块，展示 AI 如何帮助判断事件优先级、风险等级、资源匹配或调度建议。
- 输入：事件类型、位置、历史处理时长、当前资源、片区负载、紧急程度、天气或人流等上下文数据。
- 输出：AI 风险判断、调度建议、推荐负责人、预计处理时间、建议下一步动作，并标明建议原因。
- 运行方式：在事件卡片、详情页或调度面板中展示 AI 建议；可用模拟数据和静态样例完成演示。
- 验收标准：AI 功能不是装饰性文案，而是能解释“为什么这么判断”和“建议调度什么资源”；同时保留人工确认入口。

### 2.4 用最小可用版本的思路完成，不追求大而全

- 功能描述：控制功能范围，围绕核心管理场景完成一个可理解、可点击、可演示的 MVP 原型。
- 输入：核心用户角色、核心场景、少量代表性片区、少量代表性事件、关键页面结构。
- 输出：一个范围克制但闭环完整的交互原型，覆盖“看状态、找事件、看负责人、看下一步、看 AI 建议”。
- 运行方式：通过 Figma / 墨刀原型链接或设计文件预览进行演示。
- 验收标准：功能路径完整，没有大量未完成页面；演示时能在 5-10 分钟内讲清楚问题、方案和价值。

### 2.5 一页纸需求理解说明

- 功能描述：用一页纸说明对题目的理解，包括用户、场景、问题、目标、设计边界和 MVP 范围。
- 输入：题目要求、环卫调度业务理解、高管层和调度/片区主管的核心诉求。
- 输出：一页纸需求理解说明，可为 PDF、PPT 单页、设计稿页面或 README 中的单页内容。
- 运行方式：作为交付材料单独提交，或放在共享文件夹 / 压缩包中。
- 验收标准：必须控制在一页纸内，不能超页；需要清楚区分两类视角：高管视角关注“各片区运行状态、异常分布、风险等级”，调度/片区主管视角关注“事件卡在哪、谁负责、下一步做什么”。原型逻辑应形成“首页服务高管总览，点击下钻服务主管调度”的链条。

### 2.6 一页纸页面 / 功能结构图

- 功能描述：输出页面结构或功能结构，说明原型包含哪些页面、模块和跳转关系。
- 输入：核心功能模块、用户操作路径、页面层级。
- 输出：页面结构图、功能结构图、信息架构图或用户流程图。
- 运行方式：作为交付材料单独提交，或放入交付文件夹中。
- 验收标准：必须控制在一页纸内，不能超页；评审者能从结构图看懂页面组成和核心流程；图中模块名称与原型一致。

### 2.7 一个可演示的交互原型

- 功能描述：制作可点击、可走通核心流程的交互原型。
- 输入：片区数据、事件数据、状态数据、AI 建议样例、页面设计稿。
- 输出：可演示的 Figma / 墨刀原型链接或设计文件。
- 运行方式：通过原型在线访问链接或设计工具预览链接演示。
- 验收标准：至少能演示从总览进入事件详情，再查看负责人、下一步动作和 AI 建议的完整路径。

### 2.8 一段 5-10 分钟讲解说明，可现场讲，也可录屏讲

- 功能描述：准备面试讲解材料，说明需求理解、设计决策、核心流程、AI 辅助逻辑和 MVP 取舍。
- 输入：需求说明、结构图、交互原型、演示脚本。
- 输出：5-10 分钟口头讲解或录屏文件。
- 运行方式：现场打开原型演示，或提交录屏链接 / 视频文件。
- 验收标准：讲解时长控制在 5-10 分钟；重点清楚，不陷入过多细节；能回答“为什么这样设计”。

## 4. 约束条件

### 4.1 交付方式约束

- 推荐使用 Figma / 墨刀完成可演示交互原型，重点展示信息架构、页面状态和核心交互路径。
- 不建议把重点放在真实后端、真实 AI 模型、复杂技术栈或大量第三方组件上。
- 不建议依赖无法现场稳定打开的原型工具、私有环境、复杂账号权限或临时网络配置。
- 不建议只做静态海报式大屏，必须保留从总览到事件详情的可点击交互。
- AI 辅助可用模拟数据和规则样例表达，重点是展示产品判断逻辑，而不是证明算法能力。

### 4.2 交付物打包结构

- 题目原文未规定文件路径，建议用压缩包或共享文件夹统一交付。
- 推荐交付结构：
  - `README.md`：项目说明、原型访问方式、演示路径和核心设计取舍。
  - `deliverables/1.一页纸需求理解.pdf`：严格一页纸，说明用户、场景、目标和 MVP 边界。
  - `deliverables/2.页面功能结构图.pdf`：严格一页纸，说明页面组成、模块关系和跳转路径。
  - `prototype/3.交互原型访问链接.txt`：提供 Figma / 墨刀在线访问链接及必要查看说明。
  - `recording/4.原型演示录屏.mp4`：5-10 分钟演示录屏，按核心路径讲解。
  - `presentation/5.讲解提纲.pdf`：现场讲解时使用的简要提纲。
  - `assets/`：存放截图、图标、示例图片等辅助素材。

### 4.3 输入输出格式限制

- 输入建议使用少量模拟数据，不依赖真实生产接口。
- 输出必须至少包含四类材料：需求理解说明、页面 / 功能结构图、可演示交互原型、5-10 分钟讲解说明。
- 原型输出应能直观看到片区状态、事件卡片、负责人、下一步动作和 AI 辅助建议。
- 文档输出应能独立阅读，不依赖口头解释才能理解。

### 4.4 性能 / 安全 / 交互限制

- 性能：MVP 不需要承载真实海量数据，但原型页面切换和点击反馈应流畅。
- 安全：不要使用真实个人信息、真实企业内部数据或敏感位置数据。
- 交互：核心流程必须可点击、可演示，不应只有视觉稿。
- 信息展示：高管视角应优先呈现概览、风险和决策信息，避免塞入过多一线操作细节。
- AI 表达：AI 只能作为辅助判断或调度建议，不应替代人工最终决策。

### 4.5 文档交付要求

- 一页纸需求理解说明，必须严格控制篇幅。
- 一页纸页面 / 功能结构图，必须严格控制篇幅。
- 可演示交互原型，建议使用 Figma / 墨刀链接。
- 5-10 分钟讲解说明或录屏。
- README 说明项目目标、原型访问方式、演示路径和核心设计取舍。

## 5. 最可能丢分点

1. 只做了漂亮大屏，没有体现“事件卡在哪、谁负责、下一步做什么”。
2. 片区状态没有明确分级，评审者无法快速判断哪里异常。
3. AI 模块只是写了“AI 推荐”几个字，没有判断依据、调度建议或人工确认机制。
4. 原型不可点击，无法演示完整流程。
5. 功能做得太大太散，没有体现 MVP 取舍。
6. 没有一页纸需求理解，导致评审者看不出设计思考。
7. 说明文档或结构图超过一页纸，未体现高度概括和提炼信息的能力。
8. 页面 / 功能结构图和实际原型不一致。
9. 事件详情缺少负责人、位置、状态、下一步动作中的任一关键字段。
10. 演示讲解超时或没有主线，无法在 5-10 分钟内讲清价值。
11. 使用了过多真实系统假设，例如真实后端、实时地图、复杂算法，导致基础交付不完整。

## 6. 最小可交付版本

时间不够时，至少完成以下内容，保证项目像一个完整交付物：

1. 一个管理看板首页：展示 4-6 个片区的状态、关键指标和异常数量，服务高管总览视角。
2. 一个事件列表或事件卡片区：展示事件位置、类型、状态、优先级和负责人。
3. 一个事件详情交互：点击事件后展示“在哪、谁负责、下一步做什么”，服务调度/片区主管视角。
4. 一个轻量 AI 辅助模块：给出风险等级、推荐动作和 1-2 条原因解释。
5. 一份一页纸需求理解：说明用户、问题、目标、双角色视角和 MVP 范围。
6. 一份一页纸页面 / 功能结构图：说明首页、事件详情、AI 建议之间的关系。
7. 一条完整演示路径：从总览发现异常，到打开事件详情，再查看 AI 建议和下一步动作。
8. 一段 5-10 分钟讲解脚本或录屏：按“背景问题 -> 设计目标 -> 核心流程 -> AI 价值 -> MVP 取舍”讲清楚。

MVP 的核心判断标准是：评审者即使不看任何实现过程，也能通过文档和原型理解这是一个面向环卫集团的事件调度与管理看板，且能看出“高管总览 -> 主管下钻调度 -> AI 辅助判断”的完整业务链条。
